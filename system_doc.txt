# Sentinel — Complete System Documentation

> Author: Backend Team · Last updated: 2026-05-13
> Stack: Node.js · Express · SQLite (better-sqlite3) · Socket.io · Vanilla JS

---

## 1. What Sentinel Is

Sentinel is a real-time AI fraud detection layer that sits on top of a Squad merchant account. It receives every payment event via Squad's webhook, runs it through a 3-stage scoring engine, and automatically acts on the result (approve / flag / block + refund). A live dashboard shows all transactions and KPIs in real time via WebSocket.

## 5. Squad API Integration

### 5.1 Authentication

All calls use **Bearer token authentication**:
```
Authorization: Bearer <SQUAD_API_KEY>
```

The base URL is selected dynamically based on the key prefix:
```js
key.startsWith('sandbox_') → https://sandbox-api-d.squadco.com
else                        → https://api-d.squadco.com
```

### 5.2 Endpoints Used

#### `GET /account/balance`
- **Used for:** Verifying a merchant's API key on the setup screen.
- **How:** If the call returns HTTP 200, the key is valid. 401/403 = bad key.
- **Extracts:** `data.merchant_account_details.account_name` or `data.account_name` as the merchant's business name.
- **Function:** `verifyMerchantKey(apiKey)` in `squad-client/api.js`
- **Returns:** `{ valid: true, business_name: string, environment: 'sandbox'|'live' }`

#### `GET /transaction/verify/:ref`
- **Used for:** Confirming an AMBER transaction's real status with Squad before acting.
- **Called:** After a transaction scores AMBER, as a fire-and-forget background check.
- **Function:** `verifyTransaction(transactionRef)`
- **Returns:** Full Squad transaction object

#### `POST /transaction/refund`
- **Used for:** Issuing a full refund on RED (confirmed fraud) transactions.
- **Body:**
  ```json
  { "transaction_ref": "...", "refund_type": "full", "amount": 5000000 }
  ```
- **Function:** `refundTransaction(transactionRef, amount)`
- **NEW — Partial refund:**
  ```json
  { "transaction_ref": "...", "refund_type": "Partial", "amount": 2500000 }
  ```
- **Function:** `partialRefundTransaction(transactionRef, amount)`

#### `GET /transaction`
- **Used for:** Historical import — pulling last 30 days of transactions on first merchant login.
- **Query params:** `start_date` (YYYY-MM-DD), `limit`, `page`
- **Each transaction includes:** `is_suspicious` (bool) — Squad's own internal risk flag
- **Function:** `getTransactionHistory(days = 30)`
- **Returns:** Array of raw Squad transaction objects

#### `GET /dispute`
- **Used for:** Loading the disputes panel on the dashboard.
- **Function:** `getDisputes()`
- **Returns:** Array of dispute objects from Squad

#### `POST /dispute/merchant/challenge`
- **Used for:** Submitting AI-generated evidence to fight a chargeback.
- **Body:** `{ "transaction_ref": "..." }`
- **Function:** `challengeDispute(transactionRef)`

#### `PATCH /transaction/cancel/recurring`
- **Used for:** Permanently cancelling a stored card token on RED fraud detection.
- **Body:** `{ "token_ref": "..." }`
- **Function:** `cancelRecurringToken(tokenRef)`

### 5.3 Webhook — `POST /webhook/squad`

Squad sends a signed HTTP POST to this URL for every payment event.

**Signature validation:**
```
Header: x-squad-encrypted-body
Value:  HMAC-SHA512(rawBody, SQUAD_WEBHOOK_SECRET).toUpperCase()
```
Sentinel recomputes the signature and rejects the request if it doesn't match. In demo mode (`x-demo-mode: true` header), signature validation is skipped.

**Squad's real webhook payload shape:**
```json
{
  "Event": "charge_successful",
  "TransactionRef": "TXN_ABC123",
  "Body": {
    "transaction_ref": "TXN_ABC123",
    "amount": 5000000,
    "email": "customer@email.com",
    "created_at": "2026-05-13T10:00:00Z",
    "is_suspicious": false,
    "payment_information": {
      "card_bin": "420320",
      "first_6digits": "420320"
    }
  }
}
```

**Demo/simulate payload shape:**
```json
{
  "event": "charge_successful",
  "data": {
    "transaction_ref": "SIM-...",
    "amount": 25000000,
    "email": "emeka.obi74@gmail.com",
    "card_bin": "411111",
    "transaction_date": "2026-05-13T10:00:00Z",
    "is_suspicious": false
  }
}
```

## 6. Webhook Processing Pipeline (`webhook/receiver.js`)

Every incoming webhook goes through 8 steps:

```
Step 1: HMAC-SHA512 signature validation (skip in demo mode)
Step 2: Parse JSON body — normalise to common fields
Step 3: Deduplicate — check if ref already in DB, return 200 if yes
Step 4: BIN lookup — enrich card_bin with bank/brand/type/country
Step 5: Score transaction — 3-stage AI engine (see Section 7)
Step 6: Persist to SQLite
Step 7: Act based on tier:
        GREEN  → nothing
        AMBER  → verifyTransaction() + trackAmberAndAlert()
        RED    → refundTransaction() + sendFraudAlert()
Step 8: Emit 'new_transaction' via Socket.io to all connected dashboards
Step 9: Return HTTP 200
```

## 7. AI Scoring Engine

### 7.1 Overview

Every transaction is scored 0–100. The score comes from three stages added together:

| Stage | Max Score | Method |
|-------|-----------|--------|
| 1 — Rule Engine | 70 | 10 hand-crafted fraud rules |
| 2 — Z-Score Anomaly | 30 | Statistical deviation from customer history |
| 3 — ML Model | 20 | Weighted feature scoring from trained model |
| **Total** | **100** | Capped at 100 |

**Tier thresholds:**
- `0–30` → **GREEN** (safe, approved)
- `31–70` → **AMBER** (suspicious, flagged for review)
- `71–100` → **RED** (high risk, auto-blocked + refunded)

### 7.2 Stage 1 — Rule Engine (`ai-engine/rules.js`)

Ten rules, each returns a `score` and a `reason` code:

| Rule | Signal Code | Score | Triggers When |
|------|------------|-------|---------------|
| R01 | `AMOUNT_SPIKE` | 30 | Amount > 3× merchant average |
| R02 | `HIGH_VALUE_NEW` | 25 | Amount > ₦500,000 AND first-time email |
| R03 | `OFF_HOURS` | 20 | Between 1–4am West Africa Time |
| R04 | `HIGH_VELOCITY` | 35 | ≥3 transactions from same email in 5 min |
| R05 | `BIN_PATTERN` | 40 | Same card BIN used by ≥5 different emails in 60 min |
| R06 | `BEHAVIOUR_MISMATCH` | 15 | Customer always paid <₦100k, now paying >₦300k |
| R07 | `FIRST_TIME_PAYER` | 10 | No prior transaction history for this email |
| R08 | `ROUND_AMOUNT` | 15 | Amount is exactly divisible by 1,000,000 kobo |
| R09 | `FOREIGN_CARD` | 20 | Card BIN issued outside Nigeria (country ≠ NG) |
| R10 | `UNKNOWN_BIN` / `PREPAID_CARD` | 10/15 | BIN not in database, or card type = PREPAID |

Stage 1 total is capped at **70**.

### 7.3 Stage 2 — Z-Score Anomaly Detection

Uses the customer's full payment history to detect statistical outliers.

```
amounts = all past payments by this email
avg     = mean(amounts)
stdDev  = standard deviation(amounts)
z       = (current_amount - avg) / stdDev

stage2Score = min(30, max(0, z × 10))
```

If z > 2 (amount is >2 standard deviations above mean): adds `STAT_ANOMALY` signal.
If no history exists: flat score of 15 (unknown baseline = moderate risk).

### 7.4 Stage 3 — ML Model (`ml/model.json`)

Two models are trained in Python and exported as JSON — an **Isolation Forest** (unsupervised) and a **Random Forest** (supervised). RF feature importances are used as the scoring weights because supervised importances are more reliable.

**Critical fix — StandardScaler normalisation:** All features are normalised to mean=0, std=1 before training AND at runtime. Without this, `amount` (range 0–50M kobo) would dominate Euclidean distance and make all other features irrelevant. Scaler parameters are exported to `model.json` and applied in `scorer.js`.

**Feature Importances (Random Forest):**
| Feature | Importance | What it measures |
|---------|-----------|-----------------|
| `velocity` | **40.6%** | Transactions from this email in last 5 min |
| `bin_count` | **23.0%** | Distinct emails using this BIN in last 60 min |
| `amount_vs_avg` | **19.3%** | Current amount ÷ merchant average |
| `amount` | 11.3% | Raw transaction amount in kobo |
| `is_first_time` | 4.8% | 1 if no prior history, else 0 |
| `hour` | 1.0% | Hour of day in WAT (0–23) |

**Scoring at runtime (Probability Calibration):**
Instead of crude buckets, Stage 3 produces a true, continuous fraud probability (0–100%) by evaluating a polynomial fitted to the Random Forest's actual `predict_proba` output (0.987 correlation).

```
// 1. Scale each feature
scaled[f] = (rawValue[f] - scaler.mean[f]) / scaler.std[f]

// 2. Weighted sum using RF feature importances
rawWeighted = sum(scaled[f] × rfImportance[f])
normalised  = max(0, rawWeighted / MAX_WEIGHTED)

// 3. Continuous probability via polynomial calibration
fraudProb   = clip(a*x² + b*x + c, 0, 1)
stage3Score = round(fraudProb × 20)  // Contributes 0–20 pts

if (fraudProb > 0.7) trigger ML_HIGH_RISK
if (fraudProb > 0.4) trigger ML_MEDIUM_RISK
```

**Evaluation Metrics — Isolation Forest (primary, cite this to judges):**

Confusion Matrix:
- True Negatives (normal, correctly cleared): 798
- False Positives (normal, wrongly flagged): 2
- False Negatives (fraud, missed): 2
- True Positives (fraud, correctly caught): 198

| Metric | Score | What it means in fraud context |
|--------|-------|-------------------------------|
| **Accuracy** | 99.6% | 996 of 1,000 transactions correctly classified |
| **Precision** | 99% | When model flags fraud, it is right 99% of the time. Only 2 innocent customers wrongly blocked. |
| **Recall** | 99% | Of 200 real fraud cases, 198 were caught. Only 2 fraudsters slipped through. |
| **F1 Score** | **0.99** | Harmonic mean of precision and recall — the primary summary metric |

**Evaluation Metrics — Random Forest (supervised cross-check):**

| Metric | Score | Note |
|--------|-------|------|
| Accuracy | 100% | On training data — RF memorises its training set |
| Precision | 100% | On training data |
| Recall | 100% | On training data |
| F1 | 1.00 | Expected for RF on training data — not an honest generalisation metric |

> Always cite the Isolation Forest F1: 0.99 to judges. It is unsupervised — trained without labels — so the evaluation against labels is a genuine test with no leakage.

**The precision-recall trade-off in fraud detection:**
- High precision = rarely block a legitimate customer (false positives cost merchant trust)
- High recall = rarely let a fraudster through (false negatives cost money)
- 0.99 on both simultaneously is the goal. This model achieves it.

## 8. BIN Lookup (`bin-lookup/`)

**What it does:** Maps the first 6 digits of a card number (the BIN — Bank Identification Number) to card metadata.

**Data source:** `binlist-data.csv` — a 26MB CSV with ~339,000 BIN records. Loaded into a `Map` at startup for O(1) lookups. Falls back to `bins_slim.json` (Nigerian BINs only) if the full CSV isn't present.

**Fields returned:**
```js
{
  brand:      'VISA',          // card network
  type:       'DEBIT',         // CREDIT / DEBIT / PREPAID
  category:   'CONSUMER',
  bank:       'First Bank',    // issuing bank name
  country:    'NG',            // ISO alpha-2 country code
  is_nigerian: true            // derived: country === 'NG'
}
```

**Used by:** `webhook/receiver.js` (before scoring), `server.js /api/history/import` (in importer loop), `server.js /api/bin/:bin` (direct REST lookup for frontend).

### `POST /api/history/import` — How It Works

```
1. Call Squad GET /transaction with start_date = 30 days ago
2. For each transaction in the response:
   a. Normalise field names (Squad naming is inconsistent)
   b. Skip if ref already in DB (deduplication)
   c. BIN lookup on card_bin
   d. Run through scorer.scoreTransaction()
   e. Save to DB with source: 'historical', is_suspicious from Squad's field
3. Return { imported, skipped, flagged, blocked, days }
```

## 11. Email Alerts (`notifications/email.js`)

Uses **Nodemailer** with Gmail SMTP. Requires a Google App Password (not your regular Gmail password).

### RED Alert — `sendFraudAlert(txn)`
Fires immediately when a transaction scores RED. Sends a styled HTML email showing:
- Transaction reference, amount, customer email, card details, time
- AI score, all fraud signal codes
- Auto-sent, no merchant action needed

### AMBER Burst Alert — `trackAmberAndAlert(txn)`
Tracks a rolling 10-minute window of AMBER transactions. If ≥3 AMBER transactions arrive within 10 minutes, sends a single digest email listing all of them. Has a 10-minute cooldown to prevent spam.

**Both emails are fire-and-forget** (`.catch(console.error)`) — they never block the webhook response.

### Signal Codes and Display

Signal codes are stored as strings (e.g. `HIGH_VELOCITY`). The `RSN` map translates them to human readable descriptions. The `sigClass()` function assigns display severity:
- `sig-c` (red) — critical signals: HIGH_VELOCITY, ANOMALY_DETECTED, GEO_MISMATCH, ML_HIGH_RISK, ROUND_AMOUNT, UNKNOWN_BIN, FOREIGN_CARD
- `sig-w` (amber) — warning signals: AMOUNT_SPIKE, OFF_HOURS, FIRST_TIME_PAYER, PREPAID_CARD, etc.
- `sig-i` (neutral) — informational

Simulates realistic transactions by posting directly to `/webhook/squad` with header `x-demo-mode: true`.

Three functions — `simulateRed()`, `simulateAmber()`, `simulateGreen()` — each pick from pre-defined pools:

| | GREEN | AMBER | RED |

| Amounts (kobo) | ₦1,200 – ₦40,000 | ₦55,000 – ₦150,000 | ₦200,000 – ₦500,000 |

- 15% chance → RED
- 15% chance → AMBER
- 70% chance → GREEN

The KPI sidebar has manual RED/AMBER/GREEN buttons for on-demand simulation during a demo.

## 15. ML Model (`ml/`)

### Files
- `train_model.py` — Trains Isolation Forest + Random Forest, exports `model.json`
- `generate_data.js` — Generates `training_data.csv` with 5 distinct fraud patterns
- `training_data.csv` — 1,000 synthetic transactions (800 normal, 200 fraud)
- `model.json` — Exports: scaler params, both model metrics, RF importances, IF thresholds

### Training Data — 5 Fraud Attack Patterns
| Pattern | ~Count | Key signals |
|---------|--------|-------------|
| A — High-value card testing | 70 | Large amount, 1–4am, first-timer |
| B — BIN stuffing | 101 | 6+ different emails per BIN |
| C — Velocity attack | 35 | 5+ transactions from one email in 5 min |
| D — Account takeover | 30 | Known customer, 6× amount spike |
| E — Noise (hard cases) | ~remaining | Fraud that looks nearly normal |

### How model.json is used at runtime
Sentinel does NOT run Python at runtime. `scorer.js` reads `model.json` once at startup and:
1. Loads `scaler.mean` and `scaler.std` to normalise each feature
2. Loads `random_forest.feature_importances` as weights
3. For every transaction: scales features → weighted sum → normalise → threshold to 0/10/20 pts

This avoids any Python or scikit-learn runtime dependency.

## 16. is_suspicious — Dual-Layer Confidence

Every transaction from Squad's API includes `is_suspicious: boolean` — Squad's own internal risk flag from their fraud systems.

Sentinel stores this in the DB and exposes it on every transaction object. The interpretation:

| Squad `is_suspicious` | Sentinel Tier | Meaning |
|----------------------|--------------|---------|
| `false` | GREEN | ✅ Both systems clear — high confidence safe |
| `false` | AMBER / RED | ⚠️ Sentinel caught something Squad missed |
| `true` | GREEN | ⚠️ Squad flagged it, Sentinel didn't — needs review |
| `true` | RED | 🚨 Both agree — near-certain fraud |

This is Sentinel's strongest differentiator: **we catch fraud Squad's own engine misses, and when both agree, confidence is bulletproof.**

## 17. Data Flow — Full Transaction Lifecycle

```
1. Customer pays on merchant's Squad checkout
2. Squad fires webhook → POST /webhook/squad
3. Sentinel validates HMAC signature
4. Extracts: ref, amount, email, card_bin, is_suspicious, timestamp
5. Checks DB: already seen this ref? → 200 OK (skip)
6. BIN lookup: card_bin → { bank, brand, type, country, is_nigerian }
7. Scoring:
   Stage 1: Run R01–R10 → stage1Score (max 70) + reason codes
   Stage 2: Z-score vs email history → stage2Score (max 30)
   Stage 3: ML feature weights → stage3Score (max 20)
   Total = min(100, stage1 + stage2 + stage3)
8. Tier: 0–30 GREEN / 31–70 AMBER / 71–100 RED
9. Save to SQLite (ref, email, amount, card_bin, bin_info, score, tier,
                   reasons, features, timestamp, action_taken, source, is_suspicious)
10. Act:
    AMBER → Squad verifyTransaction() [background] + AMBER burst email tracking
    RED   → Squad refundTransaction() [background] + sendFraudAlert() email
11. Socket.io emit('new_transaction', fullObject) → all connected dashboards
12. Dashboard: new row slides in, KPIs update, chart nudges, toast appears
    RED: notification bell rings, alert center updates
13. Return HTTP 200 to Squad
```

## 18. ngrok (Webhook Tunnelling)

During development, Squad's servers cannot reach `localhost:3000`. ngrok creates a public HTTPS tunnel:

```
ngrok http 3000
```

This gives a URL like `https://abc123.ngrok.io`. You paste this + `/webhook/squad` into Squad's dashboard webhook settings:

```
https://abc123.ngrok.io/webhook/squad
```

Squad will then send real payment events to your local machine. Every ngrok session generates a new URL — update Squad's webhook URL when it changes.
